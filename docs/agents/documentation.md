# Documentation

When adding or changing features:

1. **Update feature documentation**: Keep the root [README.md](../../README.md) accurate for **what the app is**, **who it is for**, **prerequisites**, **limitations** (with a link to [product-audience.md](../product-audience.md) for non-goals), and short pointers (including **[getting-started.md](../getting-started.md)** for build/run, first launch, screen-dump CLI, and extended operator tips). Put **preset/show/palette JSON**, **`appsettings.json` structure**, and **NuGet dependency versions** in [docs/configuration-reference.md](../configuration-reference.md) when those change. Use **ADRs** and **`specs/`** (The Spec) for contracts; stubs under `docs/visualizers/` and `docs/ui-spec-*.md` only point to `specs/`. For ADR status and **superseded** handling, follow [docs/adr/README.md](../adr/README.md) (“For agents”); non-ADR audit notes live under [docs/adr/audits/](../adr/audits/). **New source paths**: follow [project-structure/AGENTS.md](project-structure/AGENTS.md) for the target project.

2. **Visualizer specs**: When adding or changing visualizers, update the canonical spec under **`specs/text-layers-visualizer/`** (hub: [specs/text-layers-visualizer/spec.md](../../specs/text-layers-visualizer/spec.md); per-layer: `specs/text-layers-visualizer/layers/<slug>/spec.md`). Follow [specs/TEMPLATE.md](../../specs/TEMPLATE.md). See [docs/visualizers/README.md](../visualizers/README.md) (stub index).

3. **UI specs**: When changing console layout, header, modals, or any documented screen, update the canonical spec under **`specs/console-ui/`** (hub: [specs/console-ui/spec.md](../../specs/console-ui/spec.md); see stubs under `docs/ui-spec-*.md` for links). Regenerate screenshot and line reference when the number or meaning of lines changes. Use screen dump (Ctrl+Shift+E or `--dump-after N` / `--dump-path`) to capture ASCII output for specs. Rules: `.cursor/rules/ui-specs.mdc`.

4. **README vs. deep docs vs. agents**: When user-facing behavior or run requirements change, update **README.md** (summary) and **[getting-started.md](../getting-started.md)** (step-by-step run/build/first session) as appropriate. When configuration file formats, JSON examples, or pinned dependency versions change, update `docs/configuration-reference.md`. Build, test, format, static analysis, test suite overview, and `IVisualizer`/composite-renderer notes live under `docs/agents/` (see [AGENTS.md](AGENTS.md) index)—update the relevant topic file instead of expanding the root README. **Agent steering** at repo root lives in **[AGENTS.md](../../AGENTS.md)** with links into `docs/agents/`; do not put that content in README.md (see [ADR-0068](../adr/0068-agents-md-vs-readme.md)).

5. **Document deferred work**: Every "will do later" task must be documented. When deferring work, add it to a task list in `docs/refactoring/`, a new or existing ADR, the refactoring README, or a dedicated TODO/backlog file. **Known product bugs and small TODOs** also belong in [known-bugs-and-todos.md](known-bugs-and-todos.md); keep `docs/refactoring/` for larger refactors. Do not leave deferred work only in code comments or conversation.

Do not complete feature work without updating the relevant documentation.
